# План: Переход на SSO-only авторизацию (VK + Яндекс)

## 1. Общая концепция

Система полностью переводится на авторизацию только через внешние SSO-провайдеры — VK и Яндекс. Собственные регистрация и вход по логину/паролю удаляются.

После успешной аутентификации через VK или Яндекс пользователю присваиваются внутренние идентификаторы:

- **uid** — неизменяемый числовой идентификатор, генерируется один раз и никогда не меняется
- **ckey** — строковый идентификатор (никнейм), который пользователь может изменить в будущем

Внутренняя схема (uid, ckey, сессии, токены) полностью сохраняется. Меняется только способ аутентификации: вместо ASP.NET Identity PasswordSignIn — внешний OAuth.

Дата рождения (возраст) читается из профиля VK/Яндекса при каждом входе. Какой возраст указан у провайдера — такой и в системе. Если провайдер не отдаёт дату рождения — пользователь не может завершить регистрацию.

## 2. Провайдеры

### VK ID (VK OAuth)
- Стандартный OAuth 2.0
- Запрашиваемые scope: email, birthdate
- Идентификатор пользователя: числовой VK id
- После авторизации получаем: id, email, дату рождения

### Яндекс ID (Yandex OAuth)
- Яндекс OAuth / Яндекс ID
- Запрашиваемые scope: login:email, login:birthday
- Идентификатор пользователя: Яндекс uid (числовой)
- После авторизации получаем: uid, email, дату рождения, логин

## 3. Процесс авторизации

### Первый вход (регистрация)
1. Пользователь нажимает «Войти через VK» или «Войти через Яндекс»
2. Перенаправляется на страницу OAuth провайдера
3. После подтверждения возвращается с авторизационным code/token
4. Система проверяет, есть ли пользователь с таким ExternalId + Provider
5. Если нет — создаётся новый SpaceUser:
   - uid = новый GUID (как сейчас)
   - ckey = предлагается из логина/username провайдера, пользователь может изменить на странице создания аккаунта (с проверкой уникальности)
   - ExternalId = id из провайдера
   - Provider = "VK" или "Yandex"
   - Birthday = дата рождения из провайдера
   - Email = email от провайдера
   - EmailConfirmed = true
   - PasswordHash = null
6. Создаётся сессия (через SessionManager, как сейчас)

### Повторный вход
1. Пользователь нажимает кнопку входа
2. Авторизуется через провайдера
3. Система находит SpaceUser по ExternalId + Provider
4. Обновляет Birthday и Email из профиля провайдера (всегда актуальные данные)
5. Создаётся новая сессия

## 4. Возраст и дата рождения

- При каждом входе читаем дату рождения из профиля VK/Яндекса
- Сохраняем/обновляем поле Birthday в SpaceUser
- Возраст пользователя всегда соответствует данным провайдера
- Если провайдер не отдаёт дату рождения — регистрация блокируется, пользователь видит сообщение «Не удалось получить данные о возрасте»

## 5. Legacy endpoints и обратная совместимость

Хотя активных пользователей нет, система должна сохранить обратную совместимость на уровне API:

### Legacy авторизация через токен
- Все существующие сессии и токены продолжают работать
- SessionManager, токены, expire time — без изменений
- Все API-эндпоинты, проверяющие сессии/токены — без изменений
- Клиенты (игра, лаунчер) не требуют изменений, если они уже используют токен-авторизацию

### Legacy API endpoint для лаунчера
- В AuthApiController остаётся endpoint для авторизации по токену (текущая реализация)
- Добавляется информация, какой SSO-провайдер привязан к аккаунту (для отображения в UI лаунчера)

### IdentityServer4
- IdentityServer4 остаётся, но из него убираются flows, связанные с парольным входом (ResourceOwnerPassword)
- OIDC flow для авторизации через SSO продолжает работать
- Клиентские приложения (если такие есть) могут получать токены через SSO

### База данных
- Таблица AspNetUsers остаётся (там хранятся uid, ckey, ExternalId, Provider, Birthday)
- PasswordHash становится nullable и всегда null для новых пользователей
- Существующие записи (если есть) с PasswordHash остаются нетронутыми, но использовать их нельзя — механизм парольного входа удалён
- Миграция: добавление полей ExternalId, Provider в AspNetUsers (если их ещё нет)

## 6. Что удаляется из UI и логики

- Страница регистрации (Register.cshtml + Register.cshtml.cs)
- Страница логина по паролю (Login.cshtml + Login.cshtml.cs)
- Логика PasswordSignInAsync
- Восстановление/сброс пароля (ForgotPassword, ResetPassword)
- Подтверждение email через код (теперь email подтверждается провайдером)
- Управление паролем в настройках профиля
- ResourceOwnerPassword flow в IdentityServer4

## 7. Что остаётся без изменений

- Внутренняя система идентификации (uid, ckey, сессии)
- SessionManager, TokenManager
- Все API-эндпоинты (AuthApiController, PlayerApiController, QueryApiController)
- База данных — структура AspNetUsers, HubDbContext
- Админка, управление пользователями
- Audit логгирование (AccountLogManager)
- Hub логгирование (HubAuditLogManager)
- IdentityServer4 (кроме ResourceOwnerPassword)

## 8. Нюансы и ограничения

- **Один провайдер на пользователя**: один SpaceUser = один ExternalId + один Provider. Если у человека есть VK и Яндекс — это два разных аккаунта в системе. Union аккаунтов в данном плане не предусмотрен
- **Выбор ckey**: при первом входе пользователь попадает на страницу создания профиля, где может ввести желаемый ckey (с проверкой уникальности). По умолчанию подставляется логин/username от провайдера
- **Email**: читается от провайдера. Если провайдер не даёт email — пользователь вводит его вручную на странице создания профила
- **Отсутствие провайдера**: если у пользователя нет VK или Яндекса — он не может зарегистрироваться. Это сознательное решение
- **Смена провайдера**: в будущем можно добавить, но в первой версии — нет

## 9. Этапы реализации

1. **Добавить VK OAuth провайдера** — middleware, настройка в Startup.cs
2. **Добавить Яндекс OAuth провайдера** — middleware, настройка в Startup.cs
3. **Страница выбора SSO** — страница с кнопками VK / Яндекс вместо логина
4. **Callback-обработчики** — ExternalLoginCallback: поиск/создание SpaceUser
5. **Страница создания профиля** — выбор ckey при первом входе
6. **Возраст** — чтение Birthday из профиля провайдера при каждом входе
7. **Удалить парольный логин и регистрацию** — страницы, логика, ResourceOwnerPassword
8. **Проверить legacy endpoints** — убедиться, что все API работают по токенам
9. **Миграция БД** — добавить ExternalId, Provider в AspNetUsers (если нет)
10. **Тестирование** — unit-тесты, ручное тестирование VK/Яндекс flow
